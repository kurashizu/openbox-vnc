## Project at a glance

A two-tier monorepo. The whole project is called **openbox-vnc**; the inner `openbox-vnc/` directory is just the Cloudflare Workers frontend half of it.

| Path | Role |
|---|---|
| `./` | Host controller (Python) + container image (Docker) |
| `openbox-vnc/` | Cloudflare Workers frontend (Next.js + noVNC) |

The frontend sub-project has its own [AGENTS.md](openbox-vnc/AGENTS.md) covering Cloudflare-specific details. This file covers the host + container side and cross-cutting concerns.

## Architecture invariants

These hold by design; do not "fix" them without a clear reason.

- **Container is the source of truth for X11.** All desktop state (Firefox profile, fcitx5 config, Openbox rc.xml, pcmanfm wallpaper) lives in `/home/dev/.config/` inside the container and is regenerated by `entrypoint.sh` on every boot. Editing files there is futile.
- **Host reaches container via `host.docker.internal`.** `clipboard.py` posts to `http://host.docker.internal:6081/api/clipboard` because the container is on a bridge network.
- **The host `app.py` is a pure orchestrator, not a desktop user.** It only knows how to start/stop the container and proxy WebSocket traffic. Don't add desktop logic to it.
- **No API keys in `wrangler.jsonc` `vars`.** `vars` is for non-sensitive config; secrets go through `wrangler secret put` and are stored encrypted on Cloudflare's side. See [Secrets management](#secrets-management) below.

## Secrets management

- **Resend API key** (`RESEND_API_TOKEN`): set via `cd openbox-vnc && npx wrangler secret put RESEND_API_TOKEN`. Local dev value goes in `openbox-vnc/.dev.vars`.
- **Anything sensitive** (API keys, webhooks secrets, D1 tokens): never commit. Add to `.gitignore` and use `wrangler secret`.
- **Rotation**: if a token ever appears in plaintext in the working tree (e.g., pasted in a chat, leaked in a backup), revoke it at the provider first, then re-issue. Don't just edit the file.

## Things to avoid

- **Don't** add `--password` / `-SecurityTypes VncAuth` to `x11vnc` without first deciding what network boundary the container sits behind.
- **Don't** change the in-container user from `dev` to root. The desktop session assumes the unprivileged `dev` account (PulseAudio, fcitx5, D-Bus session all key off `$HOME`).
- **Don't** introduce new Python dependencies in `app.py` without adding them to the PEP 723 `# /// script` block at the top. `uv run` resolves from there.
- **Don't** commit `mailbox.log` or any runtime artifacts. They're covered by the `*.log` rule in the root `.gitignore`.
- **Don't** edit `xorg.conf`'s `Virtual` size without also updating the matching modesetting line in `Dockerfile` / `entrypoint.sh` â€” the X server will fail to start if the two disagree.

## Cross-component changes

If your change touches both tiers (e.g., a new API endpoint used by the frontend):

1. Add the route to `app.py` (host side).
2. Add the matching fetch in `openbox-vnc/src/app/api/.../route.ts` (frontend side, if you want a server-side proxy).
3. If the frontend is calling `app.py` directly, remember it must go through the Cloudflare Worker â€” the browser can't reach `localhost:6081`. Either expose it via the Worker or use a Workers-side fetch.
